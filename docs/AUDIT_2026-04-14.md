# Audit VintedScrap

Date: 2026-04-14

## Vue d'ensemble

Le projet actuel reste fonctionnel sur ses briques principales, mais il n'est pas encore dans un état "publication GitHub propre" pour trois raisons:

- le code applicatif reste très centralisé dans `main.py`
- des régressions sont revenues dans le layout et l'intégration plugin
- des données sensibles et des données utilisateur sont encore suivies dans Git

## Problèmes bloquants

1. Layout de la grille régressé

Le code recalcule bien le nombre de colonnes visibles, mais le rendu des annonces repart ensuite sur la constante `self.COLONNES`, ce qui réintroduit le vide à droite sur grand écran.

Références:
- `main.py:1602`
- `main.py:2634`
- `main.py:2645`
- `main.py:2648`

2. Risque de crash sur prix nul dans l'analyse de négociation

`a.reduction_pct` divise par `a.prix_actuel` sans garde. Une annonce avec prix nul ou mal parsé peut faire tomber l'analyse.

Références:
- `core/analyzer.py:194`
- `core/analyzer.py:207`

3. Secrets versionnés dans le dépôt

Le webhook Discord est actuellement commité en clair. Pour une publication GitHub, il faut considérer ce secret comme compromis et le révoquer.

Références:
- `data/plugins/discord_notify/config.json:2`

4. Données utilisateur suivies malgré le `.gitignore`

Le README indique que `data/` est généré et ignoré, mais plusieurs fichiers du dossier `data/` sont déjà suivis par Git. Le `.gitignore` n'annule pas un fichier déjà versionné.

Références:
- `.gitignore:32`
- fichiers suivis observés dans `data/`

5. Système de plugins débranché de l'application

Les modules plugin existent encore, mais l'application courante n'instancie plus `PluginManager` et ne déclenche plus de hooks applicatifs. Le sous-système est donc présent dans le dépôt sans être réellement branché à l'UI principale.

Références:
- `main.py:35`
- absence d'instanciation `PluginManager(...)` dans `main.py`

## Problèmes importants mais non bloquants

1. Encodage corrompu

Le dépôt contient beaucoup de chaînes mal encodées dans les commentaires, labels et docs. Ce n'est pas bloquant pour l'exécution, mais c'est pénalisant pour une publication propre et pour la maintenance.

2. Sécurité plugin incomplète

`IMPORTS_AUTORISES` est défini mais jamais réellement appliqué dans `valider_code`. La validation repose surtout sur quelques regex.

Références:
- `core/plugin_security.py:13`
- `core/plugin_security.py:72`

3. Stockage credentials faible

Le stockage des credentials repose sur un XOR + base64 lié au hostname. C'est de l'obfuscation, pas du chiffrement fiable.

Références:
- `core/auth.py:23`
- `core/auth.py:38`

4. Profil utilisateur partiellement régressé

`user_profile` est encore importé dans `main.py`, mais les hooks de tracking et l'affichage d'affinité ne sont plus branchés dans la version actuelle du fichier principal.

Références:
- `main.py:35`
- absence d'appels `user_profile.record_*` dans `main.py`

## Recommandations de publication

Avant mise en ligne sur GitHub:

1. Révoquer le webhook Discord exposé.
2. Retirer du tracking Git tous les fichiers de `data/`.
3. Corriger la régression de layout.
4. Ajouter une garde contre les prix nuls dans `core/analyzer.py`.
5. Réparer ou retirer temporairement le système plugin.
6. Nettoyer l'encodage UTF-8 de `README.md`, `PATCHNOTE.md` et `main.py`.
7. Publier une version restructurée plutôt que l'état actuel de `main.py`.

## Travail préparé

Une base parallèle `vnext/` a été ajoutée dans le dépôt pour préparer une version plus propre, testable et publiable sans casser l'application actuelle.
